Imagine que PM2, c'est comme un "garde du corps" pour ton serveur Node.js.

Quand tu lances ton code avec node server.js, si tu fermes ton terminal SSH ou si ton code fait une erreur et plante, le serveur s'arrête net. PM2, lui, reste ouvert en arrière-plan, surveille ton application et la relance instantanément si elle "tombe".

Voici le guide de survie pour l'utiliser sur ton instance Oracle :

1. Démarrer ton serveur pour la première fois
Au lieu de faire node server.js, tu te places dans ton dossier et tu tapes :

Bash
pm2 start server.js --name "mon-app"
L'option --name est pratique pour identifier ton projet plus tard.

2. Les commandes que tu vas utiliser tous les jours
Une fois lancé, tu ne vois plus les messages "Serveur prêt sur le port 3000". Voici comment piloter l'avion :

pm2 list : C'est ton tableau de bord. Tu vois si ton app est "online", combien de fois elle a redémarré toute seule et combien de RAM elle consomme.

pm2 logs : C'est indispensable. Ça affiche en temps réel ce que tu voyais avant dans ton terminal (tes console.log, les erreurs, etc.).

pm2 restart mon-app : À faire après chaque git pull pour que le serveur charge ton nouveau code.

pm2 stop mon-app : Pour éteindre le serveur proprement.

En cas de probleme avec pm2: 
1. Identifier le nom du processus
Tape la commande suivante pour voir ce qui tourne (ou ce qui est arrêté) :

Bash
pm2 status
Regarde dans la colonne "name". Si tu vois un nom comme postit-server ou app, c'est ce nom-là qu'il faut utiliser pour le restart.

2. Relancer proprement (La solution "Reset")
Si pm2 status ne montre rien ou que tu veux repartir sur une base saine, fais ceci depuis le dossier de ton serveur :

Bash
# Supprimer les anciens résidus (si existants)
pm2 delete all

# Lancer le serveur en lui donnant un nom clair
pm2 start server.js --name "postit-app" --update-env

# Sauvegarder pour qu'il redémarre tout seul si Oracle crash à nouveau
pm2 save

Pour utiliser Git Bash efficacement dans ton projet, voici un aide-mémoire des commandes essentielles classées par utilité.

1. Navigation et Gestion de fichiers (Base)
Puisque tu travailles sur un chemin spécifique (D:\#0-Documents\PlatformIO\...), ces commandes sont tes outils de déplacement :

cd "chemin/du/dossier" : Se déplacer dans un dossier.

Astuce : Tape cd  puis fais glisser ton dossier depuis l'explorateur Windows dans la fenêtre Git Bash pour coller le chemin automatiquement.

ls : Lister les fichiers du dossier actuel.

pwd : Afficher le chemin complet où tu te trouves.

mkdir nom_dossier : Créer un nouveau dossier.

2. Commandes Git (Versionnage)
C'est le cœur de Git Bash pour ton projet postit-server :

git status : La commande la plus importante. Elle te dit quels fichiers ont été modifiés et s'ils sont prêts à être enregistrés.

git add . : Prépare tous les fichiers modifiés pour l'enregistrement (le "stage").

git commit -m "ton message" : Enregistre tes modifications localement avec un commentaire explicatif.

git push : Envoie tes modifications sur ton dépôt distant (GitHub/GitLab).

git pull : Récupère les dernières modifications du serveur vers ton ordinateur.

git log --oneline : Affiche l'historique simplifié de tes commits.

3. Commandes Node.js / Serveur (Ton projet actuel)
Même si c'est Git Bash, tu l'utilises pour piloter ton serveur :

npm start ou node server.js : Lance ton serveur.

npm install : Installe toutes les dépendances listées dans ton package.json (indispensable si tu as supprimé node_modules).

npm install nom_du_paquet : Ajoute une nouvelle bibliothèque (ex: npm install cloudinary).

ls -a : Très utile pour voir si ton fichier .env est bien présent (il est caché par défaut).

4. Raccourcis "Vie Sauvage" (Gain de temps)
Tabulation (Touche Tab) : Complète automatiquement le nom d'un fichier ou d'un dossier. Tape les premières lettres et appuie sur Tab.

Flèche Haut / Bas : Fait défiler l'historique des commandes que tu as déjà tapées (évite de tout réécrire).

Ctrl + C : Arrête de force le serveur qui tourne ou une commande bloquée.

clear : Nettoie l'écran de la console pour y voir plus clair.

💡 Un conseil pour ton erreur actuelle :
Si tu veux vérifier que ton code est bien à jour sur ton disque avant de relancer ton serveur, tape :

Bash
git status
Si tu vois des fichiers en rouge, c'est que tes modifications (comme celles du refreshView) ne sont pas encore "validées" par Git, mais elles sont bien sur ton disque. Tu peux lancer le serveur sans crainte.
